# docs/ 문서 체계 리팩토링

이 프로젝트의 docs/ 문서 체계를 리팩토링해줘.
코드 변경 시 관련 문서가 자동으로 업데이트되도록 CLAUDE.md에 코드→문서 매핑 테이블 추가 + 폴더 구조화.

---

## 작업 순서

### Phase 1: 현황 파악
1. CLAUDE.md 읽기
2. docs/ 전체 파일 목록 확인 (`find docs/ -type f | sort`)
3. 각 문서의 역할 분류:
   - **module** — 특정 모듈/기능의 구현 상세 (파일 맵, API, DB)
   - **policy** — 횡단 비즈니스 정책 (여러 모듈에 적용되는 규칙)
   - **registry** — SSOT 레지스트리 (스키마, API 목록, 결정 기록, 마이그레이션)
   - **spec** — UI/통합 스펙 (화면 설계, 시스템 통합 설계)
   - **guide** — 개발 가이드 (프롬프트, 스크립트, 튜토리얼)
   - **archive** — 레거시 문서 (폐기된 설계, 완료된 구현 계획 — SSOT 아님)
4. 분류 결과를 테이블로 정리하여 보여줘 (파일명 | 현재 경로 | 분류 | 이동할 폴더)

### Phase 2: 폴더 구조화
docs/ 아래 6개 폴더 생성 후 `git mv`로 이동:

```
docs/
├── modules/      — 모듈별 상세 문서 (구현 내역, API/DB 명세)
├── policies/     — 횡단 비즈니스 정책 (모듈 공통 규칙)
├── registries/   — SSOT 레지스트리 (결정 기록, 마이그레이션 SQL)
├── specs/        — UI/통합 스펙
│   ├── ui/       — UI 스펙
│   └── integration/ — 시스템 통합 스펙
├── guides/       — 개발 가이드 (프롬프트, 스크립트, 하위 폴더 자유)
└── archive/      — 레거시 문서 (초기 설계, 온보딩용 — SSOT 아님)
```

규칙:
- 반드시 `git mv`로 이동 (히스토리 보존)
- 이동 후 빈 디렉토리 삭제 (`rmdir`)
- docs/ 루트에 README.md 외 파일이 남지 않도록

### Phase 3: 상호 참조 링크 업데이트
- 모든 .md 파일 내 markdown 링크를 새 상대경로로 업데이트
- CLAUDE.md에서의 참조도 `docs/새폴더/파일.md` 형식으로 변경
- grep으로 모든 `docs/` 참조를 찾아서 누락 없이 수정:
  ```
  grep -rn "docs/" --include="*.md" .
  ```

### Phase 4: CLAUDE.md 업데이트
1. 기존 문서 참조 경로를 새 경로로 업데이트
2. **문서 체계 섹션 추가** — docs/ 폴더 트리 + 각 폴더 역할 설명
3. **코드 → 문서 매핑 테이블 추가**:

```markdown
### 코드 → 문서 매핑

| 코드 경로 패턴 | 변경 시 업데이트할 문서 |
|---------------|---------------------|
| `src/...` | `docs/modules/...` |
| ... | ... |
```

이 매핑은 프로젝트의 실제 코드 구조를 분석하여 작성할 것. 주요 디렉토리/파일별로 관련 문서를 매핑.

4. **주요 문서 바로가기 테이블** — 자주 참조하는 문서 5~10개

### Phase 5: docs/README.md 생성
아래 구조로 작성:

```markdown
# docs/ 문서 인덱스

## 폴더 구조
(폴더별 역할 + SSOT 여부 테이블)

## modules/ — 모듈별 상세 문서
(파일별 링크 + 설명 + 상태)

## policies/ — 횡단 비즈니스 정책
...

## registries/ — SSOT 레지스트리
...

## specs/ — UI/통합 스펙
...

## guides/ — 개발 가이드
...

## archive/ — 레거시 문서
(SSOT 아님 경고 포함)

---

## 문서 관리 규칙
1. SSOT 원칙
2. 코드 변경 시 문서 업데이트 (CLAUDE.md 매핑 참조)
3. 새 문서 생성 시 README + CLAUDE.md 매핑 업데이트
4. 문서 폐기 시 archive/로 이동 + 폐기 사유 명시
5. 마이그레이션 SQL은 registries/migrations/에 순번 파일명으로 보관
```

### Phase 6: 링크 검증
- 모든 .md 파일의 markdown 링크가 실제 파일로 resolve되는지 확인
- 깨진 링크 수정
- 검증 결과 보고

### Phase 7: 커밋 + 푸시
- 모든 변경사항을 단일 커밋으로
- 커밋 메시지: `refactor: docs/ 문서 체계 리팩토링 — 6개 폴더 구조화 + 코드→문서 매핑`
- main에 푸시

---

## 핵심 원칙
- **정보 손실 없이 이동만** — 삭제가 아니라 재배치
- **git mv 필수** — 히스토리 보존
- **프로젝트에 맞게 분류 조정** — Phase 1에서 현황 파악 후 자동으로 적합한 분류 결정
- **빈 폴더 허용** — 해당 유형의 문서가 없어도 폴더는 생성 (향후 확장용)
- **archive 폴더는 관대하게** — 완료된 계획, 폐기된 설계, 초기 온보딩 자료 등
