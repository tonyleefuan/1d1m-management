# 발송 실패 처리 + 모니터링 UX 개선 — UI 스펙

## 전일 실패 → 다음날 자동 안내 주입 (2026-04-15 추가)

4월 N일 결과 가져오기(`/api/sending/import-results`)로 실패가 확정된 뒤, 4월 N+1일 `POST /api/sending/generate` 실행 시 해당 구독의 그날 큐 **맨 앞**에 안내가 자동 주입된다.

상품 타입별 템플릿 분기:
- **고정 메시지 상품**: `failure_retry_next` ("함께 발송") — 어제치 + 오늘치 메시지가 함께 나감.
- **실시간 메시지 상품**: `failure_retry_shift` ("기간 연장") — "밀려도 하루 1건" 원칙상 어제치는 기간 연장으로 처리, 오늘은 오늘 메시지 1건만.

공통 조건:
- `send_queues.send_date = date - 1일 AND status='failed' AND is_notice=false` 를 만족하는 구독이 대상.
- 구독당 1회 주입(`is_notice=true, day_number=0, sort_order` 최소).
- `notice_templates` 의 글로벌 템플릿(`product_id IS NULL`) 사용. 없거나 content 비면 스킵(경고 로그).
- 자동 정지(3일 연속 실패)는 해당 구독을 `live` 필터에서 제외하므로 큐 생성 자체가 없음 → 자연 종료.
- 운영 순서 전제: **발송 완료 → 결과 가져오기 → 다음날 generate**. 이 순서가 지켜지지 않으면 전일 실패가 `pending`으로 남아 감지되지 않는다.

수동 "즉시 재발송"(`/api/sending/handle-failures`) 경로는 별개로 `failure_retry_now` 템플릿을 사용한다.

## 큐 정리 정책 (2026-04-09 추가)

실패 큐는 다음 조건에서 삭제됨:
- **재발송 성공**: 새 큐(sent)가 생성되어 미해결 목록에서 제외 (기존 failed 레코드는 히스토리로 보존)
- **3일 연속 실패 → 자동 정지**: pending+failed 큐 전부 삭제 (정지됨 = 처리 완료)
- **last_sent_day 수동 변경**: pending+failed 큐 전부 삭제 (새 day부터 재생성)
- **day_adjust 상대 조정**: pending+failed 큐 전부 삭제
- **구독 취소**: pending 큐 삭제

미해결 실패 모니터링: 발송 모니터링 탭 → 상태 필터 "미해결 실패" 선택 → 위 조건으로 해결되지 않은 실패만 표시

## 변경 범위

### 1. PC 카드 크기 확대 + 실패 안내
**현재:** 한 줄 5개, 작은 카드
**변경:** 한 줄 4개, 카드 내 정보 여유 있게

**실패 없는 카드:**
```
┌─────────────────────────────┐
│ 010-2785-8940               │
│ 대기 0  ● 성공 1729  ● 실패 0│
│ ████████████████████████████ │
└─────────────────────────────┘
```

**실패 있는 카드:**
```
┌─────────────────────────────┐
│ 010-2785-8940               │
│ 대기 0  ● 성공 1729  ● 실패 286│
│ ████████████████████████████ │
│ ℹ️ 다음 발송 시 자동 재발송  [변경]│
└─────────────────────────────┘
```

### 2. 실패 처리 모달
**트리거:** PC 카드의 [변경] 버튼 클릭
**내용:**

```
┌──────────────── 실패 건 처리 ─────────────────┐
│ 010-2785-8940 — 실패 286건                     │
│                                                │
│ ○ 지금 다시 보내기                              │
│   실패한 메시지를 구글시트에 다시 추가합니다.     │
│   수동으로 발송 후 결과 가져오기를 다시          │
│   눌러주세요.                                   │
│                                                │
│ ● 다음 발송 시 함께 보내기 (기본)               │
│   실패한 메시지를 내일 발송 시 내일 메시지와     │
│   함께 보냅니다. 내일은 2일치 메시지가           │
│   발송됩니다.                                   │
│                                                │
│ ○ 밀어서 보내기                                 │
│   실패한 메시지를 내일 발송합니다. 내일 보낼     │
│   예정이던 메시지는 모레로 밀리며, 구독 종료일이 │
│   하루 연장됩니다.                              │
│                                                │
│ ○ 무시하기                                      │
│   실패한 메시지를 보내지 않고 넘어갑니다. 해당   │
│   고객은 이 Day의 메시지를 받지 못합니다.        │
│                                                │
│                          [취소]  [적용]          │
└────────────────────────────────────────────────┘
```

### 3. 버튼 상태 가이드

| 상태 | 강조 버튼 | 나머지 |
|------|----------|--------|
| 대기열 없음 | 대기열 생성 (파란 + 펄스) | 평소 스타일 |
| 대기열 있고 내보내기 안 함 | 구글시트 내보내기 (파란 + 펄스) | 평소 스타일 |
| 내보내기 완료 결과 미수거 | 결과 가져오기 (파란 + 펄스) | 평소 스타일 |
| 전부 완료 | 강조 없음 | 평소 스타일 |

**순서 안 맞는 버튼 누를 때 토스트:**
- 내보내기 안 했는데 결과 가져오기 → "먼저 구글시트 내보내기를 해주세요"
- 대기열 없는데 내보내기 → "먼저 대기열을 생성해 주세요"

### 4. 알림 템플릿 추가 (메시지 관리 > 알림 템플릿)

기존 시작 알림 / 종료 알림 옆에 3개 추가:
- 즉시 재발송 알림 (a용)
- 함께 발송 알림 (b용)
- 밀어서 발송 알림 (c용)

### 5. 시작/종료 알림 발송 (대기열 생성 시)

대기열 생성 로직에서:
- Day 1 구독 → 시작 알림 메시지를 queue 맨 앞에 별도 행으로 추가
- 마지막 Day 구독 → 종료 알림 메시지를 queue 맨 뒤에 별도 행으로 추가

## API 변경

| 메서드 | 경로 | 용도 |

| POST | /api/sending/handle-failures | 실패 건 처리 (a/b/c/d) |
